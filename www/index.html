Project Description

The EJB standard took a giant step between version 2.1 and 3.0 - the same step is (in our eyes) still missing in the JMS standard: To provide a simple to use, pure Java interface solution to remote calls... only asynchronous in this case. Even Spring has not a solution satisfying us, so we (1&1) have started to develop the MessageApi as a remedy.

We would like it to become a standard in some way (in some fits of megalomania I think it maybe even become JMS 2.0 :-). We decided to make it publicly available so everybody can take a look at what we're up to. We welcome any feedback - from feature requests to patches.

Motivation

The business code should concentrate on the message that is to be sent or what it has to do when a message is received. What technology is used, where the message is going to or coming from, how the message is transformed, etc. is to be provided by a configurable adapter layer and injected into the business code.

The idea is simple: You annotate an interface as @MessageApi. An annotation processor converts all your methods into POJOs, e.g. a method public void createCustomer(String firstName, String lastName) into a POJO with the two parameters as properties (i.e. it has a getFirstName() method), and properly annotations for JAXB. Some business code will get an proxy injected for that interface. If it calls e.g. the createCustomer message, the proxy converts the call into an instance of the POJO, serializes it using JAXB, queries a registry for the correct queue, and posts that message.
